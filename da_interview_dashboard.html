Most Frequently Asked Questions

Curated from real interviews at top companies — SQL, Python, Stats, and Behavioral rounds

▸ SQL Interview Questions

What is the difference between WHERE and HAVING? ›
WHERE filters rows before aggregation. HAVING filters after aggregation (used with GROUP BY).

-- WHERE: filters raw rows
SELECT dept, COUNT(*) FROM employees
WHERE salary > 50000
GROUP BY dept;

-- HAVING: filters aggregated result
SELECT dept, COUNT(*) FROM employees
GROUP BY dept
HAVING COUNT(*) > 10;
Interview tip: If a question says "filter groups where total sales > X", that's HAVING, not WHERE.
What is the difference between RANK(), DENSE_RANK(), and ROW_NUMBER()? ›
  • ROW_NUMBER() — assigns a unique sequential number. No ties. 1, 2, 3, 4...
  • RANK() — ties get the same rank, then the next rank skips. 1, 1, 3, 4...
  • DENSE_RANK() — ties get the same rank, no skipping. 1, 1, 2, 3...

SELECT name, salary,
  ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn,
  RANK()       OVER (ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (ORDER BY salary DESC) AS drk
FROM employees;
When to use each: ROW_NUMBER for pagination / deduplication. RANK/DENSE_RANK for leaderboards where ties should be acknowledged.
Write a query to find the second highest salary. ›
Multiple valid approaches:
-- Method 1: Subquery (most common answer)
SELECT MAX(salary) FROM employees
WHERE salary < (SELECT MAX(salary) FROM employees);

-- Method 2: DENSE_RANK (handles ties correctly)
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) AS rnk
  FROM employees
) t WHERE rnk = 2;

-- Method 3: LIMIT/OFFSET (simple, risky with ties)
SELECT DISTINCT salary FROM employees
ORDER BY salary DESC LIMIT 1 OFFSET 1;
Best answer in interview: Use DENSE_RANK and mention it handles ties, which the other methods don't.
What is a CTE and when would you use it over a subquery? ›
CTE (Common Table Expression) is a named temporary result set defined with WITH. It exists only for the duration of the query.

Use CTEs when:
  • You need to reuse the same logic more than once in the query
  • Your subquery logic is complex enough to need a clear name
  • You're building a multi-step transformation (step 1 → step 2 → step 3)
  • Readability matters more than micro-performance

Subqueries are fine for simple one-off filters. CTEs are better for anything more complex because they're readable and maintainable.
Recursive CTEs can traverse hierarchical data (e.g., org charts) — a bonus point to mention if relevant.
How would you calculate a 7-day rolling average in SQL? ›
SELECT
  date,
  revenue,
  AVG(revenue) OVER (
    ORDER BY date
    ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
  ) AS rolling_7day_avg
FROM daily_revenue
ORDER BY date;
Key concept: ROWS BETWEEN 6 PRECEDING AND CURRENT ROW defines the window frame as the current row plus 6 prior rows = 7 days total. Use RANGE BETWEEN instead of ROWS BETWEEN if you need calendar-based intervals instead of row-based.
What is the difference between UNION and UNION ALL? ›
UNION combines results from two queries and removes duplicates (adds a DISTINCT step — slower).

UNION ALL combines results and keeps all duplicates (faster — no dedup step).

Interview answer: Always use UNION ALL unless you specifically need deduplication, because it's faster. If the two result sets are already unique by design, UNION is just unnecessary overhead.
You have a table of user events. Write a query to find users who logged in on two consecutive days. ›
SELECT DISTINCT a.user_id
FROM logins a
JOIN logins b
  ON a.user_id = b.user_id
  AND DATEDIFF(b.login_date, a.login_date) = 1;
Alternative using LAG():
SELECT DISTINCT user_id FROM (
  SELECT user_id, login_date,
    LAG(login_date) OVER (PARTITION BY user_id ORDER BY login_date) AS prev_date
  FROM logins
) t
WHERE DATEDIFF(login_date, prev_date) = 1;
Mention both approaches — the JOIN is classic, the LAG approach shows window function fluency.

▸ Python / pandas Questions

How do you handle missing values in pandas? Walk me through your approach. ›
Step 1: Understand the extent
df.isnull().sum()           # count nulls per column
df.isnull().mean() * 100    # % nulls per column
Step 2: Decide strategy based on type and business context
  • Drop rows: df.dropna(subset=['col']) — when < 5% missing and random
  • Fill with constant: df['col'].fillna(0) — for counts, flags
  • Fill with mean/median: df['col'].fillna(df['col'].median()) — for numeric, use median if skewed
  • Forward/backward fill: df['col'].ffill() — for time series
  • Impute with model: sklearn KNNImputer — for complex multivariate patterns
Interview tip: Always explain WHY you chose a method, not just what you ran.
What is the difference between loc and iloc in pandas? ›
  • loc — label-based: select by index label or column name. df.loc[0, 'name']
  • iloc — position-based: select by integer position. df.iloc[0, 2]

df.loc[df['city'] == 'Tampa']       # filter by label condition
df.iloc[0:5, 1:3]                   # first 5 rows, cols 1 and 2
Common gotcha: When the index is the default integer range, loc[0] and iloc[0] return the same row — but after sorting or filtering, the index labels may no longer match positions.
How do you merge two DataFrames? What are the different join types? ›
pd.merge(df1, df2, on='user_id', how='inner')   # INNER JOIN
pd.merge(df1, df2, on='user_id', how='left')    # LEFT JOIN
pd.merge(df1, df2, on='user_id', how='outer')   # FULL OUTER JOIN
pd.merge(df1, df2, on='user_id', how='right')   # RIGHT JOIN

# Different column names in each table:
pd.merge(df1, df2, left_on='id', right_on='user_id', how='left')
concat vs merge: Use pd.concat to stack DataFrames vertically (UNION). Use pd.merge for horizontal joins (JOIN).
How would you find duplicate rows in a DataFrame and remove them? ›
df.duplicated().sum()                      # count total duplicates
df[df.duplicated()]                        # view duplicate rows
df[df.duplicated(subset=['email'])]        # duplicates on specific column

df.drop_duplicates()                       # remove duplicates, keep first
df.drop_duplicates(subset=['email'], keep='last')  # keep last occurrence
Interview tip: Ask what defines a "duplicate" in business context — is it a full-row match, or just a duplicate on a key like email or user_id? The answer changes your strategy.

▸ Statistics & A/B Testing Questions

What is a p-value? Explain it to a non-technical stakeholder. ›
Technical definition: The p-value is the probability of observing results at least as extreme as what we got, assuming the null hypothesis is true.

Plain English for a stakeholder: "If we assume our new button has zero effect, the p-value tells us how likely we'd see a difference this large just by random chance. A p-value of 0.03 means there's only a 3% chance the results are due to random noise."

Common misconceptions to avoid saying:
  • ❌ "p=0.03 means there's a 97% chance the treatment works"
  • ❌ "p=0.03 means the effect is large or practically meaningful"
  • ✅ "p=0.03 means statistical significance at 0.05 threshold, but we also need to assess practical significance"
How do you design an A/B test end to end? ›
Step 1 — Define hypothesis: What change are we making? What is the expected direction of effect?

Step 2 — Define success metric: Primary metric (e.g., checkout conversion rate) + guardrail metrics (e.g., revenue per user).

Step 3 — Calculate sample size: Use a power calculator (80% power, α = 0.05). Determine minimum detectable effect (MDE) based on business significance.

Step 4 — Run the test: Random assignment, equal groups, no peeking. Run for at least one full business cycle (usually 1–2 weeks).

Step 5 — Analyze results: Run t-test or chi-square. Check p-value and confidence interval. Segment results by device, cohort, and geography.

Step 6 — Make a decision: Statistical significance + practical significance + business context = recommendation.

Key gotchas to mention: novelty effect, network effects, Simpson's paradox in segments, running tests too short.
What is the difference between Type I and Type II errors? ›
  • Type I error (False Positive): We reject the null hypothesis when it's actually true. We conclude a treatment works when it doesn't. Controlled by α (significance level). α = 0.05 means 5% chance of Type I error.
  • Type II error (False Negative): We fail to reject the null when it's actually false. We miss a real effect. Controlled by β. Power = 1 - β. Typical target: 80% power (β = 0.20).

Business framing: "A Type I error means we ship a feature that doesn't actually help — wasted engineering effort. A Type II error means we discard a feature that would have helped — missed opportunity. The trade-off between these drives your choice of α and sample size."
What is the Central Limit Theorem and why does it matter for analysis? ›
The CLT states: If you take sufficiently large random samples from any population (regardless of its distribution), the distribution of sample means will be approximately normal.

Why it matters for analysts:
  • It's why t-tests and z-tests work even when the underlying data isn't normally distributed
  • It justifies using parametric statistical tests on real-world skewed business data
  • "Sufficiently large" is typically n ≥ 30

Interview answer tip: Connect it to A/B testing — "This is why we can use a t-test to compare conversion rates even though individual user behavior is binary (convert or not), as long as our sample size is large enough."

▸ Product & Business Analytics Questions

Our DAU dropped 25% overnight. How do you investigate? ›
This is the most common product analytics question. Use a structured funnel:

Step 1 — Validate the data: Is the metric drop real or a data pipeline issue? Check logging, ETL jobs, and data freshness.

Step 2 — Narrow the scope by time: When exactly did it start? Sudden drop = likely technical issue. Gradual = behavioral change.

Step 3 — Segment by dimensions:
  • Geography — one region or global?
  • Platform — iOS, Android, web?
  • User cohort — new vs existing users?
  • Feature area — which screens/actions dropped?
Step 4 — Check for external causes: App store update, server outage, competitor launch, marketing campaign end, holiday.

Step 5 — Formulate a hypothesis and test: State the most likely cause and what data would confirm or rule it out.

Step 6 — Recommend action: Fix + monitor plan, not just the diagnosis.
How would you define the key metrics for a new product feature? ›
Framework: use a 3-layer metric structure

1. Primary (success) metric: The one metric this feature is designed to move. Should be measurable, time-bound, and tied to user behavior.

2. Secondary (supporting) metrics: 2-3 metrics that confirm the primary is moving for the right reasons (not just through noise).

3. Guardrail metrics: Metrics that should NOT be harmed. Revenue, retention, NPS, latency.

Example — a new onboarding flow:
  • Primary: Day 7 retention rate
  • Secondary: Onboarding completion rate, time-to-first-value
  • Guardrail: Support ticket volume, overall conversion rate
Mention: Avoid vanity metrics (pageviews, impressions) and pick metrics you can actually act on.
What is the difference between a north star metric and a KPI? ›
North Star Metric (NSM): A single metric that best captures the core value delivered to users. It should correlate with long-term revenue and guide the whole company's direction.
Examples: Airbnb → nights booked. Spotify → time spent listening. LinkedIn → monthly active users.

KPI (Key Performance Indicator): A set of operational metrics tracked to measure progress toward specific goals. Multiple KPIs exist at team and feature level, and they ladder up toward the NSM.

The relationship: KPIs are the levers. The NSM is the flywheel outcome. A team's KPI might be "checkout completion rate" while the company NSM is "gross merchandise value."

Interview tip: Knowing this distinction signals product thinking, not just technical skill. It will impress interviewers at growth-stage or product-led companies.

▸ Behavioral Questions

Tell me about a time you found a significant insight from data. ›
STAR structure:

Situation: Set the business context — what problem or question existed?

Task: What were you asked to do (or what did you proactively investigate)?

Action: What data did you pull? What SQL / Python / tool did you use? How did you validate the insight?

Result: What was the impact? Always quantify. "This led to a 15% reduction in X" or "saved 40 analyst hours per month" or "the team prioritized feature Y as a result."

Tips:
  • Pick a story where YOUR analysis drove a decision — not just where you ran a report
  • Mention one thing that was unexpected or counterintuitive in the data
  • Show you can translate numbers into business language
Tell me about a time you had to explain a complex analysis to a non-technical audience. ›
What they're really asking: Can you translate data into clear business language? Can you manage up and build stakeholder trust?

What a strong answer shows:
  • You understood what the audience cared about (the decision, not the math)
  • You removed jargon and focused on the "so what"
  • You used a visual, analogy, or concrete example rather than raw numbers
  • The audience understood and trusted the analysis enough to act on it
Avoid: "I simplified it for them" — sounds condescending. Say instead: "I reframed the analysis around the business question they were trying to answer."
How do you handle a situation where two data sources give conflicting numbers? ›
This is a very common DA interview question. Strong structure:

1. Validate both sources: Check data freshness, granularity, and definition. Are both measuring the same thing with the same filters?

2. Trace the discrepancy: Profile both datasets. Find the overlap and the gap. Is the difference a join issue? A timezone difference? Different NULL handling?

3. Surface it — don't hide it: Tell stakeholders immediately. "I found a discrepancy of X% between source A and source B, and I'm investigating the root cause."

4. Establish a single source of truth: Work with data engineering to agree on the canonical source for each metric going forward. Document it.

Key message: Show that you prioritize data integrity over convenience, and that you communicate proactively rather than picking the number that looks better.
Where do you see yourself in 3–5 years? (For a DA role) ›
What they're really evaluating: Are you motivated? Will you grow with the company? Are you a flight risk?

Strong answer framework:
  • Short-term (1 year): Master the tools, domain, and data stack at this company. Build relationships with key stakeholders.
  • Mid-term (2–3 years): Lead analysis for a major product area or business function. Begin mentoring junior analysts.
  • Long-term (3–5 years): Senior analyst or lead role — either staying deep in analytics or moving toward analytics engineering or data science, depending on the company's structure and your growth.
Tailor to the role: If the company values technical depth, emphasize analytics engineering / ML. If it's a business-oriented role, emphasize strategic influence and stakeholder management.

🧠 Common Trick Questions & Gotchas

What is Simpson's Paradox and when does it matter in analysis? ›
Simpson's Paradox is when a trend that appears in several groups of data reverses or disappears when the groups are combined.

Classic example: Treatment A has a higher overall success rate than Treatment B. But when you split by severity (mild/severe cases), Treatment B beats Treatment A in both groups. The paradox is caused by a confounding variable (case severity) being unevenly distributed.

Why it matters in practice:
  • A/B test results can reverse when you control for user segments
  • Aggregate metrics can hide problems in specific cohorts
  • Always segment results before presenting aggregated conclusions
Mentioning this in an interview scores major points — it shows you think about confounders, not just correlations.
Correlation vs causation — give a real business example. ›
Correlation: Two variables move together. Ice cream sales and drowning rates both increase in summer — correlated, not causal.

Causation: One variable directly causes a change in the other. Requiring seatbelts causes fewer traffic fatalities.

Business example: "We noticed users who use our mobile app more than 3x per week have 60% higher lifetime value. But we can't conclude that pushing notifications to increase session frequency will raise LTV — high-frequency users may simply be the segment that was already highly engaged."

How to test for causation in business: Randomized A/B test (gold standard), quasi-experimental design (diff-in-diff, regression discontinuity), or instrumental variable analysis.

Always ask: Is there a confounding variable? Could there be reverse causation?
